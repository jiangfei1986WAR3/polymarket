## 1. 当前项目的核心目标

当前项目的主线目标没有改变，仍然是：

- 构建一套可在 `Polymarket` 上长期稳定运行的“最多 6 步马丁”自动交易系统。
- 研究层负责：
  - 64 组 `6` 位 `U/D pattern` 的回测、扫描、coverage 扫描、walk-forward 验证；
  - 条件风险表、allowed_states、72 小时爆仓概率等风险过滤；
  - 输出可直接被执行层使用的策略 bundle。
- 执行层负责：
  - 实盘鉴权、下单、成交对账、订单状态恢复、自动回款、风控暂停、daemon 化；
  - 未来通过 Windows GUI 方式长期运行。

当前项目主线仍然是：

- `Polymarket` 研究与执行系统
- 不是币安事件合约主线
- 不是旧软件/旧项目延续

## 2. 当前项目处于什么阶段

### 2.1 研究层

研究层已经不是概念验证，而是已经形成较完整闭环：

- 基于 Binance `BTCUSDT` 1H K 线建立信号/标签源。

- 已完成条件风险表与 state 过滤逻辑。
- 已完成 coverage 扫描与 walk-forward 样本外验证。
- 已具备导出策略 bundle 的能力：
  - `strategy_config.json`
  - `allowed_states.json`
  - `walk_forward_summary.json`
- 已有“全量重算并优选”流程，且 GUI 侧已接入：
  - 先刷新 Binance 数据
  - 再做质量校验
  - 再做全量扫描与推荐

当前研究层的重要共识：

- 当前系统选出的推荐策略，是“在当前这套扫描逻辑、当前收益模型、当前评分规则下”的最优策略。
- 不是绝对永远最优，但在当前逻辑下，它就是当前推荐结果。

### 2.2 执行层

执行层已经跨过“能否自动化下单”的最大门槛：

- 已成功完成 `Polymarket` 真实自动化下单验证。
- 已打通：
  - L1 / L2 鉴权
  - 正确账户模型识别
  - 正确 signer / funder 路径
  - 真实市场定位
  - 真实订单成交

当前执行层不是卡在“能不能下单”，而是卡在：

- 如何更稳定地长期运行
- 如何在不牺牲安全性的前提下继续自动化
- 如何把 GUI / daemon / 风控 / 状态恢复打磨到更稳

## 4. 当前系统已经解决并验证过的关键问题

### 4.1 自动回款路径错误已修复

之前自动回款失败的核心问题不是“主下单链路错了”，而是：

- `auto_redeem.ts` 以前走了错误的 relayer / safe 执行路径
- 但当前真实账户更接近 deposit-wallet 路径

已经完成的关键修复：

- 在自动回款前先检查 deposit wallet 是否已部署
- 改为 `executeDepositWalletBatch()` 路径
- 保留异步模型，不阻塞主流程

重要原则：

- 只修回款路径
- 不扰动已正常工作的主下单链路

### 4.2 全量优选前的数据刷新与质量校验已接入

之前全量扫描有“直接用旧 CSV”的风险，现已改成：

- GUI 点击全量重算时
- 先刷新 Binance 1H 数据
- 再做质量检查
- 最后才跑全量优选

### 4.3 walk-forward 累计回撤口径已修正

之前最大回撤口径存在低估风险。
当前已改成更接近真实的累计权益曲线回撤计算。

这件事很重要，因为它直接决定：

- 策略排名
- 推荐可信度
- 风险判断是否失真

### 4.4 当前 GUI / daemon 的基本稳定性已验证

近期多次通过运行日志验证过：

- 直接关闭窗口后重开，不会自动重复跑出双 daemon
- runtime_state、heartbeat、events log 一致性正常
- 多次“看起来不下单”的时段，本质是 `BLOCK`，不是系统卡死

也就是说：

- 当前版本已经具备“相对稳定的可运行形态”
- 但仍未达到“安全与长期运维都完全放心”的程度

## 5. 当前研究层的重要结论

### 5.1 allowed_states / coverage 很关键

近期补充研究明确说明：

- `coverage` 越大，通常意味着 allowed_states 白名单越大，交易次数越多
- 但白名单太宽时，会把边缘 state 也放进来
- 结果往往是：
  - entries 变多
  - blowup 变多
  - 回撤恶化
  - 收益反而不一定更好

已验证的结论：

- `DDUUUU`、`UUUUUD` 这类头部策略都表现出相似规律：
  - coverage 太低：太保守，收益吃不满
  - coverage 适中：收益和风险平衡较好
  - coverage 太高：blowup 和回撤先恶化，收益后变差

### 5.2 当前正式系统已经“考虑了 coverage 救活 pattern 的可能”

正式优选脚本并不是固定单一 coverage。
它本身已经：

- 对每个 pattern 扫多组 coverage
- 在当前规则下从中挑 best coverage

所以：

- 某个 pattern 原本在 `c10` 亏损
- 换成 `c30 / c50 / c70` 转正
- 正式系统理论上是能选到这种情况的

只是正式系统不会显式告诉用户：

- “这个 pattern 是被 coverage 救活的”

而 `analyze_pattern_coverage_grid.py` 做的，就是把这种现象单独显式展示出来。

### 5.3 爆仓后条件风险研究有价值，但不是当前主线

已讨论过“爆仓分布、爆仓后会不会进入更危险阶段”等问题。

当前结论：

- 有研究价值
- 但不能直接作为主开仓逻辑
- 更适合作为未来二级风险研究
- 当前优先级低于：
  - 邮箱自动化安全方案
  - 执行层稳定性
  - 正式实盘链路完善

- 正式系统不往币安执行层迁移
- 后续若再讨论币安，只作为独立研究话题，不影响当前主线

## 7. 重大安全事件与根本原因判断

### 7.1 发生过什么

这段时间出现了非常严重的安全问题：

- Polymarket 账户出现限制 / close-only
- 出现异常 `Withdrawal -$18.52`
- OKX 钱包出现用户本人未操作的转账记录
- 后续又出现异常登录验证码邮件

这已经不能再按“只是 Polymarket 后台显示问题”看待。

### 7.2 当前最保守判断

当前必须把旧钱包 / 旧环境视为高风险：

- 旧外部钱包不再可信
- 旧服务器环境也不能默认完全可信
- 不能继续拿旧钱包私钥直接跑自动化

即便没有 100% 法证证据证明“代码被改”，也必须按更保守的安全原则行事。

### 7.3 这件事对项目方向的影响

它直接改变了后续自动化的优先级：

- 以前重点是“怎么让系统更自动化”
- 现在必须变成：
  - 如何在继续自动化时显著降低私钥泄露风险
  - 如何把资金、账户、环境做隔离

## 8. 关于邮箱账户自动化的当前共识

这是当前最关键的新方向。

### 8.1 已确认的官方结论

基于官方帮助文档与开发文档，已经确认：

- `Polymarket` 支持 `Google / Email / Crypto Wallet` 注册与登录
- 邮箱账户底层并不是“没有钱包”，而是有 wallet / proxy 模型
- 邮箱账户可以导出私钥
- 一旦导出私钥，这把私钥理论上可以用于程序化签名和自动化交易

所以：

- “邮箱账户不能自动化”这个说法不准确
- 更准确的说法是：
  - 邮箱账户默认更适合人工登录体验
  - 但导出私钥后，仍然可以进入自动化路径

### 8.2 安全上必须如何理解

不能把邮箱账户导出的私钥理解成：

- 只是站内下注权限
- 不能提现
- 泄露后只要邮箱验证码没丢就没事

更安全的默认假设应该是：

- 这仍是一把应按“热钱包级别权限”对待的高风险私钥
- 一旦泄露，就应视为该账户已失去安全信任

### 8.3 为什么邮箱账户仍然更适合未来自动化

虽然权限不能低估，但邮箱账户路线相较于继续用旧外部钱包，仍然更合适：

- 可以新注册、专用、隔离
- 可以小额、可废弃
- 可以不和主钱包、旧钱包、OKX 钱包混用
- 即便出问题，影响范围更可控

### 8.4 当前最重要的架构判断

以后若继续自动化，推荐的不是：

- 继续用旧钱包私钥

而是：

- 新建专用邮箱账户
- 导出其私钥
- 只作为 Polymarket 小额自动化热账户使用

## 9. 当前软件与“邮箱账户自动化”的兼容判断

### 9.1 当前软件不是“只能钱包模式”

当前系统从代码结构上看，并不是死绑某种钱包登录 UI。
它底层依赖的是：

- `privateKey`
- `walletAddress`
- `funderAddress`
- `signatureType`
- `relayerApiKey`

所以从工程结构看：

- 它本质上更像“signer 配置系统”
- 而不是“只能用某个钱包入口”

### 9.2 这意味着什么

如果切到邮箱账户模式，主下单链路不一定要推翻重做。
更合理的做法是：

- 保留当前钱包模式
- 增加一个邮箱账户配置模式

也就是软件支持：

- `wallet`
- `email_proxy`

两套模式并存。

### 9.3 推荐的后续改法

建议做成：

- 新增 `accountMode`
- GUI 可切换：
  - 钱包模式
  - 邮箱账户模式
- 底层仍使用统一 signer 配置

这样可以：

- 不破坏当前钱包配置
- 以后如果又想回到钱包模式，仍然可用
- 邮箱模式和钱包模式互不干扰

### 9.4 需要特别注意的点

虽然主下单链路大概率只需改配置和少量兼容层，但：

- 自动回款 / relayer / proxy 模式兼容
- 仍需要做一次完整验证

也就是说：

- 纯下单：更可能是“小改”
- 下单 + 自动回款 + 完整链路：仍需要专门测试

## 10. 当前最值得继续做的事情

### 10.1 主优先级：邮箱账户自动化兼容

当前最重要、最现实、最符合安全需求的下一步，就是：

- 不推翻现有系统
- 让现有系统兼容邮箱账户自动化模式

推荐实施顺序：

1. 保留钱包模式
2. 新增邮箱账户配置模式
3. 用测试邮箱账户先验证：
   - 读账户
   - 派生 API key
   - 查市场
   - 下小额测试单
4. 再验证自动回款链路

### 10.2 次优先级：保持当前正式系统稳定

当前不要做的事：

- 不要让研究脚本侵入正式执行链路
- 不要为了币安事件合约去动主系统
- 不要在旧钱包基础上继续扩展实盘自动化

### 10.3 后续可继续讨论的研究项

这些可以继续研究，但不应影响主线：

- `analyze_pattern_coverage_grid.py` 的深入解读
- coverage 与 allowed_states 的进一步微调逻辑
- 爆仓后条件风险分布研究

## 11. 当前最重要的“不要忘记”的判断

为了避免以后换机后又重复走弯路，这里把几条最关键判断单独写死：

- 当前 `Polymarket` 自动化系统已经具备真实下单能力，不再是“概念验证”。
- 当前研究层已经能选出“在当前逻辑下的最优策略”。
- 当前主系统仍然是 `Polymarket`，不是币安事件合约。
- 研究脚本必须和正式实盘系统明确隔离。
- 当前最大的阻碍已经不再是“技术上能不能下单”，而是“如何更安全、更稳定地长期运行”。
- 旧钱包 / 旧安全环境不能继续默认信任。
- 邮箱账户自动化是当前最值得继续推进的新方向。
- 邮箱账户导出的私钥应按高权限热钱包私钥看待，不能低估其权限。
- 正确方向不是“去掉钱包模式”，而是“在当前系统里增加邮箱账户模式，并与钱包模式并存”。

## 13. 一句话总结

当前项目已经不是“能不能做 Polymarket 自动化交易”的问题，而是：

**如何在已经具备自动化交易能力的基础上，把系统继续推进成一个更安全、更稳定、兼容邮箱账户模式、并且不会因研究分支而污染当前实盘主线的正式系统。**
