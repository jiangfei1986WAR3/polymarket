# Polymarket 马丁系统（开发文档 / 当前进展说明）

日期：2026-05-12

> 说明：本仓库之前的“旧软件/旧项目”已停止继续开发。
> 当前后续开发聚焦于 **Polymarket 的马丁交易系统**（含策略研究、风险过滤、以及未来的 Windows GUI 实盘工具）。

---

## 1. 项目目标与边界

### 1.1 目标

构建一套可在 Polymarket 上运行的“马丁型（最多 6 步）”自动交易系统，核心能力包含：

- 使用 Binance `BTCUSDT` 1H K 线作为信号/标签源（`close > open` 视为 Up，否则 Down）。
- 在“理想 1:1 赌局”假设下完成大规模回测/风险统计：
  - 穷举 6 位 Up/Down pattern（共 `2^6 = 64` 种）；
  - 统计总盈亏、最大回撤、6 连亏（爆仓）事件与间隔分布。
- 构建“倒着推”的 **条件风险表**：
  - 以“过去 6 根 1H K 的形态（64 种 state）”为条件，统计未来 `N=72` 小时内发生爆仓的概率；
  - 采用 train/test 切分，验证样本外效果；
  - 用覆盖率（如 60%）生成允许开仓的 state 集合（allowed_states）。
- 未来实盘（Polymarket）：
  - 手动入金；软件只负责交易；
  - 采用“每周滚动更新一次”的方式（训练窗口最近 365 天），持续同步市场结构变化。

### 1.2 边界与当前简化

当前研究与回测均采用简化假设（用于研究“爆仓分布/风险过滤”的有效性）：

- **理想 1:1**：押对盈利 = `stake * 1.0`，押错亏损 = `stake`。
- 暂不计入：手续费、点差、滑点、成交不确定性、Polymarket 价格（YES/NO 份额价格）等。
- 所有策略当前均以“每小时开一局”做抽象：`close > open` 判定 Up。

这些简化对“马丁爆仓概率、爆仓间隔、风险过滤是否有样本外价值”仍具参考意义；
但要落地到 Polymarket 实盘，后续需要把收益模型替换为“按订单簿价格成交后的实际 PnL”。

## 2. 代码结构与关键模块

主要代码在：`kronos_v1_backtest/src/kronos_v1_backtest/`

### 2.1 数据模块

- `data/binance_downloader.py`
  - 从 Binance REST API 下载 1H K 线，写入 CSV。
  - CSV 字段：`open_time, open, high, low, close, volume`。

- `data/binance_csv.py`
  - 读取 CSV 并产出 `Candle` 列表。

- `data/quality_check.py`
  - 校验重复K线、缺失K线、OHLC 合法性。

### 2.2 马丁回测（核心）

- `martingale/enumerate_patterns.py`
  - `MartingaleConfig`：pattern_len=6、base_stake=2、max_steps=6、payout_b、fee_rate。
  - `backtest_pattern(...)`：对单个 pattern 做逐小时模拟：
    - 赢：盈利 `+stake * payout_b`（扣 fee_rate），level 重置到 1；
    - 输：亏损 `-stake`，level + 1；
    - 连输至第 6 步仍输：记一次 `level6`（爆仓事件），重置回第 1 步。
  - 输出指标：
    - `pnl_u`：总盈亏
    - `max_drawdown_u`：最大回撤
    - `n_level6_losses`：爆仓次数
    - `level6_loss_gaps`：相邻爆仓事件间隔（以小时K计）
  - `backtest_all_patterns(...)`：穷举 `2^6=64` 个 pattern。

### 2.3 条件风险表（倒着推）

- `martingale/conditional_risk.py`
  - `conditional_risk_by_state(...)`：对每个过去 6 根形态（64 state）统计：
    - `p(level6 within next N hours)`：未来 N 小时内至少爆一次的概率
    - `p(gap==6 within next N hours)`：未来 N 小时内“连续爆仓”的概率
    - `TTF`（time-to-first）分布的均值/中位数：第一次爆仓距离当前多少小时
  - 采用 contiguous train/test 切分（默认按 start_i 序列前 2/3 train 后 1/3 test，可由 CLI 参数控制）。
  - 当前默认 N=72（未来 3 天窗口），用于风险过滤研究。

### 2.4 “状态驱动马丁”实验（结果：无稳定优势）

- `martingale/state_driven.py`
  - 实验策略：每次启动一轮马丁时，用“启动时刻之前的最近 6 根K线形态”直接作为 6 步 pattern。
  - 两种映射：
    - `forward`：step1 用最老（t-6），step6 用最新（t-1）
    - `reverse`：step1 用最新（t-1），step6 用最老（t-6）
  - 一年数据回测结果（理想 1:1、base=2U、6 步）：
    - forward：`pnl_u = -744`（亏损）
    - reverse：`pnl_u = +20`（接近打平）
  - 结论：该思路整体更像“无 edge 的噪声”，对映射细节敏感，不建议作为主策略。

## 3. CLI 使用方式（研究/回测工具）

所有命令建议使用 `PYTHONPATH` 直接运行（无需安装包）：

### 3.1 下载一年数据

下载 BTCUSDT 1h 最近 365 天：

```bash
PYTHONPATH=kronos_v1_backtest/src \
  python3 -m kronos_v1_backtest.cli \
  --download-binance --symbol BTCUSDT --days 365 \
  --csv kronos_v1_backtest/data/raw/binance/BTCUSDT_1h_365d.csv
```

### 3.2 穷举 64 个 pattern 的 6 步马丁回测

```bash
PYTHONPATH=kronos_v1_backtest/src \
  python3 -m kronos_v1_backtest.cli \
  --csv kronos_v1_backtest/data/raw/binance/BTCUSDT_1h_365d.csv \
  --martingale-enum --topn 10
```

输出：TopN by PnL、TopN by fewest level6 等。

### 3.3 条件风险表（倒着推）

以 pattern=UUUUUU（每一步都押涨）为例，统计“过去 6 根 state → 未来 72 小时爆仓概率”。

```bash
PYTHONPATH=kronos_v1_backtest/src \
  python3 -m kronos_v1_backtest.cli \
  --csv kronos_v1_backtest/data/raw/binance/BTCUSDT_1h_365d.csv \
  --conditional-risk --risk-horizon 72 --train-ratio 0.75 \
  --risk-pattern UUUUUU
```

输出包含：
- baseline（train/test 加权平均爆仓概率）
- 64 个 state 的 train/test 概率与 TTF 统计

### 3.4 状态驱动马丁实验

```bash
PYTHONPATH=kronos_v1_backtest/src \
  python3 -m kronos_v1_backtest.cli \
  --csv kronos_v1_backtest/data/raw/binance/BTCUSDT_1h_365d.csv \
  --state-driven --state-mapping forward
```

可切换 `--state-mapping reverse`。

## 4. 已完成的关键研究结论（当前共识）

### 4.1 穷举 pattern 选“历史最优”并不能保证未来优势

- 6 位 pattern 只有 64 种，历史上总会有“碰巧更好”的组合。
- 但若缺乏可预测优势（edge），这种选择更接近数据挖掘/过拟合。

### 4.2 马丁不会创造 edge，只改变收益分布

- 典型形态：多数时间小赚（每轮 +2U），少数时间大亏（爆一次 -126U）。
- 因此风险度量（爆仓次数、爆仓间隔、最大回撤）比“总盈亏”更关键。

### 4.3 “倒着推风险过滤”有可能作为风险管理模块

我们做了更严格的验证方式：

- 仅用 train 的统计表选 allowed_states（目标覆盖率，如 60%）；
- 固定 allowed_states，直接在 test 样本外评估。

在一年数据中（pattern=UUUUUU、N=72、train_ratio=0.75），覆盖率从 10% 到 90% 的扫描表明：

- 覆盖率越低：爆仓更少，但机会也更少（总利润不一定最大）。
- 覆盖率越高：逐渐接近“不过滤”的 baseline。
- 在此次样本中，覆盖率约 60% 的点在 test 上表现出较好的“风险/收益折中”。

## 5. 实盘系统（Polymarket）规划与当前决策

### 5.1 入金策略

- **入金手动完成**（用户自行从交易所/钱包入金到 Polymarket）。
- 软件不负责自动入金/提币，以降低安全风险与工程复杂度。

### 5.2 鉴权与下单方式（Polymarket.com）

当前共识：采用官方推荐的 CLOB 鉴权流程（L1 私钥签名 → L2 三件套），并使用 SDK 完成订单签名/请求头。

安全策略：
- **不在硬盘持久化保存私钥**（每次启动手动输入，模式 1）。
- 软件仅在内存中持有：私钥/签名器、L2 三件套。
- 软件异常退出/服务器重启后，需要人工重新解锁继续运行。

### 5.3 更新节奏

- **每周更新一次** allowed_states：
  - 训练窗口：最近 365 天；
  - N=72；
  - 目标覆盖率：建议从 60% 起步，后续可参数化并做 walk-forward 观察。

### 8.2 Tab 1：配置（Config）

**A. 账户/鉴权（Polymarket.com / CLOB）**
- 输入框：热钱包私钥（仅本次会话；默认隐藏；可切换显示）
- 按钮：解锁钱包
- 文本：钱包地址（解锁后显示）
- 文本：链/网络（Polygon mainnet / chain_id=137）
- 按钮：生成/派生 CLOB API 凭证（L2 三件套）
- 文本：CLOB API key 状态（已生成/未生成，仅显示状态，不显示 secret）
- 按钮：连接测试
- 文本：测试结果（成功/失败原因）

**B. 策略参数（Risk Gate + Martingale）**
- 下拉：标的（默认 BTC 1H）
- 输入：未来窗口 N（默认 72 小时）
- 输入：覆盖率（默认 60%，范围 10%~90%）
- 输入：训练窗口（默认 365 天）
- 输入：更新频率（默认每周一次；选择周几/几点）
- 输入：基础下注 base_stake（默认 2U）
- 输入：最大步数 max_steps（固定 6；先不开放）
- 下拉：pattern（默认 UUUUUU；后续可扩展）
- 按钮：立即重算 allowed_states（仅在“已解锁 + 已生成凭证”或离线研究模式下可用）
- 文本：allowed_states 数量、训练覆盖率/测试覆盖率（若有）

**C. 风控参数（强制）**
- 输入：单日最大亏损（达到则暂停开仓）
- 输入：最大连续爆仓次数（达到则暂停）
- 输入：最大持仓/最大未平仓订单数
- 复选：网络异常时暂停开仓
- 复选：下单失败连续 N 次暂停

**D. 数据源参数（Binance K 线）**
- 输入：CSV 路径（可选）
- 按钮：下载最近 365 天 1H 数据
- 按钮：质量检查
- 文本：数据范围、缺失/重复统计

## 7. 风险提示（必须写入项目约束）

- 马丁策略存在“小赢多次、偶尔大亏”的结构性风险。
- 任何基于历史统计的 allowed_states 都可能在市场 regime 切换后失效。
- 私钥在内存中仍存在被盗风险；建议使用热钱包、小资金运行。
