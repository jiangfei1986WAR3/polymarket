# Polymarket 项目换服务器恢复指南

日期：2026-05-17

> 用途：
> - 这份文档用于在“更换服务器 / 新装系统 / 迁移到新电脑”时，快速恢复当前项目。
> - 它重点解决的是：源码怎么拉、环境怎么补、哪些内容能恢复、哪些内容必须手工补回。
> - 建议与 `polymarket_martingale_handoff_2026-05-15.md`、`polymarket_martingale_system_dev_doc.md` 一起阅读。

---

## 1. 先说结论

当前项目可以恢复，而且恢复难度总体不高，但要区分三件事：

- 恢复源码：容易
- 恢复到可继续开发 / 回测：也比较容易
- 恢复到和旧服务器一模一样的运行状态：不一定

原因很简单：

- 已推送到 GitHub 的源码、研究脚本、策略输出、开发文档，可以直接拉下来
- 但一些本地文件为了安全没有入库，因此不会自动跟着仓库回来

这属于正确做法，不是缺点。

---

## 2. 当前已经在 GitHub 上的内容

只要远程仓库还在，下面这些通常都可以恢复：

- 项目源码
- `martingale_research/` 研究与执行相关代码
- 研究脚本、测试脚本
- 已导出的策略输出
- 开发文档
- 阶段总结文档
- 恢复 / 续接类文档

也就是说：

- 换服务器后，项目的“知识、逻辑、代码结构、研究成果”大部分都能恢复

---

## 3. 当前不会自动恢复的内容

以下内容之前就是故意不推到 GitHub 的：

- `runtime_state/`
- `build_dist/`
- `app_config.json`
- `node_modules/`

这些文件不入库是合理的，因为它们通常包含：

- 本机路径
- 本地运行痕迹
- 缓存 / 打包产物
- 潜在敏感配置
- 无法跨机器直接复用的依赖目录

这意味着：

- 新服务器拉完源码后，不能指望它立刻拥有旧机器的运行状态
- 需要重新安装依赖、重建配置、重新生成打包产物

---

## 4. 换服务器前建议先准备什么

如果旧服务器还在，建议迁移前先做一次人工整理。

建议至少保留以下信息：

- GitHub 仓库地址
- 当前使用的分支名
- 当前稳定版本对应的提交号
- Python 版本
- Node.js 版本
- 当前运行方式
- 是否启用了 GUI / daemon / ts_executor
- 当前使用的钱包模式还是未来邮箱模式

建议额外单独备份但不要上传到公开仓库的内容：

- `app_config.json`
- 任何私钥、API key、relayer key
- 运行日志中有价值的排障信息
- 如果确实需要延续状态，再单独备份 `runtime_state/`

重要原则：

- 凭据文件只做离线保管
- 不要把私钥直接提交到 GitHub
- 不要把带本机路径的运行文件误传到仓库

---

## 5. 新服务器恢复的大致流程

建议按下面顺序执行。

### 5.1 拉取源码

```bash
git clone https://github.com/jiangfei1986WAR3/polymarket
```

然后进入项目目录：

```bash
cd polymarket
```

---

### 5.2 优先阅读这几份文档

先不要急着直接运行，先读文档确认当前项目状态：

- `README.md`
- `polymarket_server_recovery_guide.md`
- `polymarket_martingale_handoff_2026-05-15.md`
- `polymarket_martingale_phase_summary_2026-05-13.md`
- `polymarket_martingale_system_dev_doc.md`

这样做的原因是：

- 项目已经不只是一个脚本集合
- 现在有研究层、执行层、GUI、daemon、安全边界、research-only 脚本边界
- 不先看文档，很容易恢复错方向

---

### 5.3 恢复 Python 环境

当前仓库默认以 `Python 3.12` 为主。

推荐流程：

```bash
cd polymarket\martingale_research
python -m venv .venv
.venv\Scripts\activate
python -m pip install --upgrade pip
pip install -r requirements.txt
```

如果后续需要直接使用源码入口，通常还需要确保：

- 当前目录在 `martingale_research/`
- 或正确设置 `PYTHONPATH=src`

---

### 5.4 恢复 Node.js 环境

如果需要 GUI、执行器、打包能力，继续恢复 `ts_executor` 相关依赖。

典型流程：

```bash
cd polymarket\martingale_research\ts_executor
npm install
```

说明：

- `node_modules/` 不会从旧服务器自动恢复
- 必须在新服务器重新安装

---

### 5.5 恢复本地配置

这是换服务器最关键的一步之一。

因为 `app_config.json` 没有入库，所以你需要：

- 从旧服务器手工复制一份安全版配置
- 或按当前系统要求重新填写一份

通常需要确认的配置包括：

- 私钥
- `walletAddress`
- `funderAddress`
- `signatureType`
- `relayerApiKey`
- 运行模式相关配置

如果未来切到邮箱账户模式，原则也一样：

- 仍然需要把导出的私钥等必要参数提供给本地软件
- 只是账户来源从外部钱包变成邮箱账户 / proxy 模式

---

### 5.6 重新生成本地运行目录和打包目录

以下内容一般不需要从旧服务器复制：

- `node_modules/`
- `build_dist/`

这类内容通常应在新机器重新生成。

而 `runtime_state/` 要不要恢复，要看你的目标：

- 只是继续开发 / 回测：通常不用恢复
- 只是重新部署运行：多数情况下也可以从空状态重新开始
- 想尽量延续旧机器的运行现场：可以额外人工迁移

建议默认策略：

- 先不迁移 `runtime_state/`
- 优先在新服务器做一次干净启动

---

## 6. 按目标区分恢复难度

### 6.1 目标一：继续开发和研究

这是最容易恢复的场景。

只需要：

- 拉源码
- 装 Python 依赖
- 确认研究数据和脚本路径

通常就可以继续：

- 回测
- 分析
- 阅读代码
- 修改脚本
- 继续策略研究

### 6.2 目标二：恢复执行器并重新自动化运行

这个也不算难，但比“仅开发”多几步：

- 装 Node.js 依赖
- 补 `app_config.json`
- 检查凭据是否有效
- 检查网络、RPC、账户配置
- 做一次小额验证

推荐顺序：

1. 先验证只读能力
2. 再验证派生 API key
3. 再验证查市场
4. 最后才做小额下单

### 6.3 目标三：恢复成和旧服务器完全一致

这个最难，因为它不只是恢复源码，而是恢复整个运行现场。

它取决于你是否还保留：

- 旧配置文件
- 旧运行状态目录
- 旧日志
- 旧环境版本信息
- 旧凭据

如果这些都没保留，那么：

- 项目本身仍能恢复
- 但旧服务器的“精确运行现场”未必能 100% 还原

---

## 7. 推荐的恢复检查清单

新服务器恢复后，建议按下面顺序检查。

### 7.1 源码检查

- 能正常 `git clone`
- 目录结构完整
- 关键文档存在
- 关键脚本存在

### 7.2 Python 检查

- 虚拟环境可激活
- `requirements.txt` 安装完成
- 研究脚本可正常启动

### 7.3 Node.js 检查

- `npm install` 成功
- `ts_executor` 依赖齐全
- GUI / 执行器入口能正常启动

### 7.4 配置检查

- `app_config.json` 已补齐
- 地址格式正确
- 私钥格式正确
- `signatureType` 正确
- 如果启用 relayer，相关配置正确

### 7.5 运行检查

- 先做只读预览
- 再做小额测试
- 最后才接正式运行

---

## 8. 恢复时最容易忽略的问题

### 8.1 误以为 clone 完就等于恢复完成

不是。

`git clone` 解决的是源码恢复，不是环境恢复，也不是密钥恢复。

### 8.2 误把旧服务器的敏感文件重新传到 GitHub

不要为了“方便恢复”而把下面这类文件公开上传：

- 私钥
- API key
- `app_config.json`
- `runtime_state/`
- 打包产物

### 8.3 直接用旧钱包继续跑正式自动化

当前项目已经明确把旧钱包 / 旧环境视为高风险。

所以恢复后若要继续自动化，建议优先考虑：

- 新环境
- 新隔离账户
- 小额验证
- 未来兼容邮箱账户模式

### 8.4 忽略 research-only 与正式系统的边界

以下脚本是研究用途，不应当被当成正式执行入口：

- `martingale_research/scripts/evaluate_event_contract_85.py`
- `martingale_research/scripts/analyze_pattern_coverage_grid.py`
- `martingale_research/scripts/compare_whitelist_layers.py`

恢复时要避免把这些研究脚本误当成正式实盘启动方式。

---

## 9. 对当前项目的实际判断

按照目前仓库状态：

- 恢复源码：没有大问题
- 恢复开发上下文：也没有大问题
- 恢复研究与回测能力：难度较低
- 恢复自动化执行能力：可行，但取决于本地配置和凭据

换句话说：

- 以后如果更换服务器，我可以帮你把云端源码拉下来
- 然后按文档把环境、依赖、配置一步步补齐
- 恢复到“继续开发”和“重新运行”这两个层面，整体都是可控的

---

## 10. 一句话总结

当前项目的 GitHub 仓库已经足够支撑“换服务器后恢复源码与继续开发”，而要恢复成可重新自动化运行的系统，只需要在新机器上补齐环境、依赖与本地配置；真正不能自动随仓库恢复的，主要是出于安全考虑故意不入库的状态文件和敏感配置。
